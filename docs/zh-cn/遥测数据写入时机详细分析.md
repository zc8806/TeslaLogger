# TeslaLogger 遥测数据写入数据库时机详细分析

## 一、整体架构

TeslaLogger 遥测数据写入流程遵循 **"接收 → 解析 → 分类处理 → 条件写入"** 模式：

```
特斯拉车辆
    │ Fleet Telemetry 推送（TLS）
    ▼
fleet-telemetry 服务器（Go）
    │ ZeroMQ PUB  tcp://192.x.x.x:5284
    ▼
TelemetryConnectionZMQ.Run()  [接收]
    │ Thread.Sleep(100ms) 轮询
    │ ReceiveMultipartMessage()
    ▼
TelemetryParser.handleMessage(json)  [解析]
    │
    ├─ 遍历字段 key
    │    ├─ PackVoltage/PackCurrent → InsertBatteryTable
    │    ├─ CruiseState → InsertCruiseStateTable
    │    ├─ Gear/ChargeState → handleStatemachine [状态判断]
    │    ├─ Location/Odometer/Speed → InsertLocation
    │    ├─ ACChargingPower/Soc → InsertCharging
    │    └─ OutsideTemp/TPMS → InsertStates
    │
    ▼ [条件写入]
 各数据表（pos/battery/charging/cruisestate/alerts/TPMS）
```

---

## 二、核心写入方法与时机

### 1. InsertBatteryTable → `battery` 表

#### 触发字段：
- PackVoltage（电池组总电压）
- PackCurrent（电池组电流）
- BrickVoltageMin/Max（单体电压极值）
- ModuleTempMin/Max（模组温度极值）
- IsolationResistance（绝缘电阻）
- LifetimeEnergyUsed（累计使用能量）

#### 写入条件：
⚠️ **必须同时满足两个条件**：
1. 接收到至少一个电池相关字段
2. **PackVoltage 和 PackCurrent 同时到达**（避免单字段写入）

#### 写入逻辑：
```csharp
// 源代码位置：TelemetryParser.cs:1550-1730
if (字段是电池相关 && PackVoltage已获取 && PackCurrent已获取)
{
    构造 INSERT SQL，包含所有已更新字段
    ON DUPLICATE KEY UPDATE 已存在则更新
    ExecuteNonQuery()
}
```

---

### 2. InsertCruiseStateTable → `cruisestate` 表

#### 触发字段：
- CruiseState（巡航状态）

#### 写入条件：
- 接收到 CruiseState 字段
- 且与上次值不同（避免重复写入）

#### 状态映射：
| 原始值 | 存储值 |
|--------|--------|
| Off | 0 |
| On | 1 |
| Override | 2 |
| Standby | -1 |
| Standstill | -2 |
| 其他 | -99（未知状态） |

---

### 3. handleStatemachine → 状态机控制（触发其他写入）

#### 核心功能：
判断车辆状态（行驶/充电），控制 `Driving`、`acCharging`、`dcCharging` 三个布尔标志。

#### 状态转换逻辑：

| 状态 | 进入条件 | 退出条件 |
|------|---------|---------|
| **行驶中** | Gear != "P" 或 VehicleSpeed > 0 | Gear == "P" 或 60分钟无行驶 |
| **交流充电** | ChargeState == "Enable" 且 PackCurrent > 2A | Gear != "P" 或 Speed > 0 或 DetailedChargeState == Stopped |
| **直流充电** | FastChargerPresent == true 且 PackCurrent > 5A 或 DCChargingPower > 5kW | FastChargerPresent == false 或 PackCurrent < 2A 或 Speed > 0 |

#### 触发的关键写入：
- **开始行驶** → `InsertFirstPos()` → 写入 pos 表第一行
- **开始充电** → `StartACCharging()` / `StartDCCharging()` → 触发 InsertFirstCharging()

---

### 4. InsertLocation → `pos` 表（轨迹数据）

#### 触发字段：
- Location（GPS坐标）
- Odometer（里程）
- VehicleSpeed（车速）

#### 写入条件：
⚠️ **严格条件**：
- 必须在 `Driving == true` 状态下
- 接收到 Location 字段

#### 写入逻辑：
```csharp
// 源代码位置：TelemetryParser.cs:1280-1331
if (Driving == true && 有Location字段)
{
    解析经纬度、里程、车速
    car.DbHelper.InsertPos(lat, lng, speed, odometer, ...)
    → INSERT INTO pos (CarID, Datum, lat, lng, speed, odometer, ...)
}
```

> **重要**：静止时只更新内存状态（CurrentJSON），不写入 pos 表！

---

### 5. InsertCharging → `charging` 表

#### 触发字段：
- ACChargingEnergyIn（交流充电累计能量）
- ACChargingPower（交流充电功率）
- DCChargingEnergyIn（直流充电累计能量）
- DCChargingPower（直流充电功率）
- Soc（当前电量）
- IdealBatteryRange（理想续航）
- RatedRange（标定续航）

#### 写入条件：
⚠️ **必须满足**：
- 车辆处于充电状态（`IsCharging == true`）
- 接收到至少一个充电相关字段

#### 写入逻辑：
```csharp
// 源代码位置：TelemetryParser.cs:1108-1158
if (IsCharging == true && 有充电字段)
{
    构造 INSERT SQL，包含所有已更新字段
    ON DUPLICATE KEY UPDATE 已存在则更新（处理重复推送）
    ExecuteNonQuery()
}
```

---

### 6. InsertStates → 多目标写入

#### 目标1：`pos` 表（温度补充）

**触发字段**：OutsideTemp/InsideTemp
**写入条件**：接收到温度字段，且在行驶状态下

#### 目标2：`TPMS` 表（轮胎气压）

**触发字段**：TpmsPressureFl/Fr/Rl/Rr（四轮气压）
**写入条件**：接收到任意一个轮胎气压字段（每次单个更新）

#### 目标3：内存状态（不写入DB）

以下字段仅更新 `car.CurrentJSON`（用于 Web UI 和 MQTT 广播）：

| 字段 | 更新时机 |
|------|---------|
| SentryMode | 接收到 SentryMode 字段 |
| PreconditioningEnabled | 接收到 PreconditioningEnabled 字段 |
| BatteryHeaterOn | 接收到 BatteryHeaterOn 字段 |
| VehicleName | 接收到 VehicleName 字段（不同时更新） |
| Trim | 接收到 Trim 字段（不同时更新） |
| CarType | 接收到 CarType 字段（不同时更新） |
| Version | 接收到 Version 字段（不同时更新） |
| DoorState | 接收到 DoorState 字段 |
| Locked | 接收到 Locked 字段 |
| *Window | 接收到 Window 字段 |

---

### 7. InsertAlert → `alerts` 表

#### 触发条件：
- 接收到包含 `"alerts"` 关键字的消息（顶层 key）

#### 消息格式：
```json
{
  "vin": "LRWXXXXXXXXXXX",
  "alerts": [
    {
      "name": "VCFRONT_a184",
      "startedAt": "2024-01-01T10:00:00Z",
      "endedAt": "2024-01-01T10:05:00Z",
      "audiences": ["Driver", "Mechanic"]
    }
  ],
  "createdAt": "2024-01-01T10:00:00Z"
}
```

#### 写入逻辑：
```csharp
// 源代码位置：TelemetryParser.cs:1390-1477
foreach (alert in alerts)
{
    查找或创建 alert_names 记录（nameID）
    INSERT INTO alerts (CarID, startedAt, nameID, endedAt)
    ON DUPLICATE KEY UPDATE endedAt = ...
    
    foreach (audience in audiences)
    {
        INSERT INTO alert_audiences (alertsID, audienceID)
        ON DUPLICATE KEY IGNORE
    }
}
```

## 三、特殊写入机制

### 1. 数据写入的条件性总结

| 表名 | 写入条件 |
|------|---------|
| `pos` | Driving == true（行驶中） |
| `charging` | IsCharging == true（acCharging \|\| dcCharging） |
| `battery` | PackVoltage AND PackCurrent 同时到达 |
| `TPMS` | 任意一个轮胎气压字段到达（单个更新） |
| `cruisestate` | CruiseState 字段到达且值变化 |
| `alerts` | 接收到 alerts 消息 |

---

### 2. 内存聚合与批量写入

各方法先将字段值聚合在内存变量中，满足条件后一次性写入，避免单字段频繁写 DB：

```csharp
// TelemetryParser 类字段（内存存储）
private double? lastPackVoltage;      // 上次电压
private double lastPackCurrent;       // 上次电流
private double lastSoc;              // 上次电量
private double? lastOdometer;        // 上次里程
private double? lastOutsideTemp;     // 上次外部温度
...
```

---

### 3. ON DUPLICATE KEY UPDATE 幂等性

`charging` 表和 `alerts` 表使用此语义：
```sql
INSERT INTO charging (...) VALUES (...)
ON DUPLICATE KEY UPDATE field1=@field1, field2=@field2
```

确保重复推送的同一时刻数据不会产生重复行。

---

### 4. 状态超时机制

**行驶状态超时**（60分钟无行驶）：
```csharp
public bool Driving
{
    get
    {
        if (driving)
        {
            var ts = DateTime.Now - lastDriving;
            if (ts.TotalMinutes > 60)
            {
                driving = false;
                Log("Stop Driving by timeout 30 minutes ***");
            }
        }
        return driving;
    }
}
```

**在线状态超时**（10分钟无消息）：
```csharp
public bool OnlineTimeout()
{
    var ts = DateTime.UtcNow - lastMessageReceived;
    return ts.TotalMinutes > 10;
}
```

## 四、数据流时序图

```
接收到遥测消息（JSON）
    │
    ├─ handleMessage() 解析顶层 key
    │
    ├─ "data" 字段
    │    ├─ 遍历 data[] 中每个 {key, value} 对
    │    │
    │    ├─ 字段类型判断
    │    │    ├─ 电池字段 → 更新内存变量
    │    │    ├─ 巡航字段 → 更新内存变量
    │    │    ├─ 状态字段 → 调用 handleStatemachine()
    │    │    ├─ 位置字段 → 更新内存变量
    │    │    ├─ 充电字段 → 更新内存变量
    │    │    └─ 其他字段 → 更新 CurrentJSON
    │    │
    │    └─ 条件判断（是否满足写入条件）
    │         ├─ 电池表：PackVoltage && PackCurrent 同时到达
    │         ├─ 巡航表：值变化
    │         ├─ 位置表：Driving == true
    │         ├─ 充电表：IsCharging == true
    │         └─ 其他表：字段到达
    │
    └─ "alerts" 字段
         └─ 直接写入 alerts 表和 alert_audiences 表
```

## 五、与 REST API 轮询模式的对比

| 对比项 | 遥测推送模式（Fleet Telemetry） | REST API 轮询模式 |
|--------|-----------------------------|------------------|
| **数据方向** | 车辆主动推送 → 服务器订阅 | 服务器主动拉取 → 车辆响应 |
| **写入时机** | 字段到达时条件判断写入 | 轮询间隔到达时批量写入 |
| **pos表写入** | 仅行驶时（Driving=true） | 每次轮询都写入 |
| **充电表写入** | 仅充电时（IsCharging=true） | 每次轮询都写入 |
| **频率控制** | 字段配置采样间隔（1s-600s） | 固定轮询间隔（5s-30s） |
| **网络依赖** | 车辆在线即可，无需持续连接 | 需要持续访问 Tesla 服务器 |

## 六、总结

TeslaLogger 遥测数据写入的核心设计理念是 **"状态驱动 + 条件控制"**：

1. **分层解耦**：传输层（ZMQ）与业务解析层（TelemetryParser）分离
2. **字段路由**：按 JSON key 分类路由到对应处理方法
3. **条件写入**：严格的状态判断，避免无效数据写入
4. **内存优化**：字段值先聚合在内存，满足条件后一次性写入
5. **幂等性保障**：ON DUPLICATE KEY UPDATE 处理重复推送

这种设计确保了数据库写入的合理性和高效性，同时避免了因遥测字段到达时间差导致的数据不一致问题。
